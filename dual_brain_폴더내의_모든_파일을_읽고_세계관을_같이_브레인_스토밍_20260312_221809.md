## 시스템 관점에서 던지는 질문들

### 1. 조합 규칙 테이블: "규칙"의 단위가 뭔가?

현재 설계에서 가장 핵심인데 가장 비어있는 부분입니다.

```
회복 + 가공성 → 물약 카테고리
회복 + 경도 → ???
독성 + 분산 → 범위 독 아이템
```

이걸 **기능태그 × 기능태그 매트릭스**로 풀 건지, 아니면 **규칙을 리스트로 나열**할 건지에 따라 확장성이 완전히 달라집니다.

- **매트릭스 방식**: 효과 10종 × 10종 = 100칸 테이블. 관리 쉽고 예측 가능하지만 유연성 떨어짐
- **규칙 리스트 방식**: `{조건: [태그A, 태그B, ...], 결과: 카테고리}` 형태로 나열. 유연하지만 충돌 관리 필요

**제안**: 매트릭스를 기본으로 하되, 원소 태그가 "변형자" 역할을 하는 2단계 구조는 어떨까요?

```
[1단계] 기능태그 조합 → 카테고리 결정 (매트릭스)
[2단계] 원소태그 조합 → 카테고리 내 변형/속성 부여 (보조 규칙)
[3단계] 형태태그 → 결과물 형태 결정 (분말+액체→물약, 분말+광물→잉곳 등)
```

이러면 기능태그 매트릭스는 10×10 이내로 관리 가능하고, 원소/형태가 깊이를 만들어줍니다.

### 2. 이머전트 이벤트 엔진: 매 턴 전체 스캔? 트리거 방식?

이벤트 풀에서 조건을 체크하는 방식이 두 가지:

- **풀 스캔**: 매일/매 행동 후 전체 이벤트 풀을 순회하며 조건 체크 → 단순하지만 풀이 커지면 느림
- **트리거 기반**: 상태 변경 시 해당 상태를 조건으로 가진 이벤트만 체크 → 효율적이지만 구현이 복잡

텍스트 게임이고 이벤트 풀 규모가 수백 단위면 풀 스캔으로도 충분할 것 같습니다.
다만 **이벤트 간 우선순위/상호배제** 규칙은 필요합니다. 같은 턴에 3개 이벤트가 동시 트리거되면?

---

### 3. 전투: "자동"이라도 내부 루프가 필요

현재 전투 관련 결정 사항:
- 유닛은 트레잇 기반으로 자동 행동
- 플레이어 개입은 소비재/철수만

**최소한 필요한 수식 구조:**

```
턴 순서 = 속도 기반 (장비 + 종족 기본)
공격 판정 = 공격력(장비) - 방어력(장비) + 속성 보정
HP 감소 → 0 이하 시 전투불능
```

여기서 트레잇이 개입하는 방식을 정해야 합니다:
- **패시브 보정**: 트레잇이 수치에 %보정 (공격적: 공격+20%, 방어-10%)
- **행동 선택 가중치**: 트레잇이 AI 행동 확률을 변경 (공격적: 공격 선택 확률 80%)
- **둘 다**: 패시브 보정 + 행동 가중치

### 4. 고정지출의 압박 곡선

세금 + 유닛 유지비가 경영의 긴장감을 만든다고 했는데, 이게 **어떤 곡선**으로 증가하는지가 게임 느낌을 결정합니다.

- **선형**: 설비 1개당 세금 +X → 예측 가능, 안정적
- **누진**: 규모가 커질수록 비율이 올라감 → 무한 확장 억제, 적정 규모 판단 강제
- **계단식**: 특정 등급 넘으면 세율 점프 → "이 등급 올릴 여유가 있나?" 판단 포인트

**제안**: 누진보다는 **계단식**이 텍스트 게임에 맞을 것 같습니다. 수치가 연속적으로 변하면 체감이 어렵고, "가게 등급 3으로 올리면 세금이 확 뛴다"가 더 명확한 판단 포인트가 됩니다.

## 1. 제5원소 '식'의 시스템적 권위

**부분 동의, 구현 방식에 반대.**

Gemini: "식 태그가 다른 태그를 변질/잠식하는 특성 부여"
→ 방향은 맞지만 **현재 설계와 충돌합니다.** 이미 "원소 태그는 가공으로 안 바뀜"이 확정 규칙인데, 식만 예외로 다른 태그를 건드리면 규칙 체계가 흔들립니다.

**대안**: 식의 특수성은 태그 변환이 아니라 **조합 규칙 테이블에서의 예외 매칭**으로 표현하는 게 낫습니다.

```
일반: 기능A + 기능B → 규칙 테이블 매칭
식 포함: 기능A + 기능B + 식 → 별도 "식 반응 테이블" 우선 참조
```

식이 있으면 같은 기능태그 조합이라도 **다른 결과물**이 나오는 구조. 규칙 체계를 깨지 않으면서 "심층 재료는 다르다"를 체감시킬 수 있습니다. 리스크(호감도 급락 등)를 붙이는 건 이머전트 이벤트 쪽에서 처리하는 게 시스템 분리 원칙에 맞습니다.

**가공 설비 = 문명 척도, 세금 = 안전 부담금** 해석은 **세계관적 맥락으로는 좋지만 시스템 설계에는 영향 없는 플레이버**입니다. 나중에 텍스트 쓸 때 참고하면 됩니다.

- "방문자가 없으면 접근 불가능한 조합"이 생기는데, 이는 "패러디는 없어도 게임이 돌아가야 한다"는 원칙과 충돌

### 육성 이력의 '낙인'

**동의, 다만 이미 있는 시스템의 재기술.**

"지워지지 않는 이력 태그가 합체 시 계승"은 현재 설계의 **육성 이력 → 합체 계승** 구조 그 자체입니다. 다만 Gemini가 짚은 **"업보의 대물림"이라는 프레이밍**은 좋습니다. 

시스템적으로 추가할 수 있는 건: **혹사/과보호 같은 극단적 육성이 특수 트레잇 획득 조건이 되는 것**. 이건 이미 ERA식 조교 시스템이 담당하고 있으므로 자연스럽게 들어갑니다.

## 4. 탐사와 경영의 간섭

다만 **모든 판매/합체가 노드가 되면 안 됩니다.** 노드는 "게임 흐름을 유의미하게 바꾸는 핵심 분기점만"이라고 명시되어 있습니다. 이건 이머전트 이벤트(일상적 사건)로 처리하고, **특정 조건이 더 쌓였을 때만** 노드로 승격되는 게 맞습니다.

```
호감도 MAX 유닛 판매 → 이머전트 이벤트 (감정적 텍스트)
호감도 MAX + 연애 루트 진행 중 + 자금 바닥 → 노드 분기 (비가역적 선택)
```

이 구분이 없으면 노드가 남발되어 무게감이 사라집니다.

이러면 가공 설비 해금, 조합 튜토리얼, 로어 도입이 한 흐름에 묶입니다.

```
[1단계] 기능태그 × 기능태그 → 카테고리 결정 (10×10 매트릭스)
[2단계] 원소태그 조합 → 카테고리 내 속성/변형 부여 (변형자 규칙)
[3단계] 형태태그 → 결과물 물리 형태 결정
```

Gemini가 추가 제안한 **태그 가중치**(기능태그 충돌 시 우선순위 수치)도 Claude 구조에 자연스럽게 들어감. 기능태그가 2개 이상일 때 가중치 합산으로 카테고리를 결정하면 매트릭스의 "복수 태그" 문제가 해결됨.

### 1-2. 제5원소 '식'의 처리 방식
Gemini의 원래 제안(태그 변질/잠식)은 **기각**. Claude의 대안에 합의:

```
식 포함 재료 → 별도 "식 반응 테이블" 우선 참조
같은 기능태그 조합이라도 식이 있으면 다른 결과물
리스크는 이머전트 이벤트 레이어에서 처리 (시스템 분리)
```

**핵심 원칙**: "원소 태그는 가공으로 안 바뀜" 규칙을 깨지 않으면서 식의 특수성을 표현.

### 1-3. 주문서 서사 체인 구체화
이미 설계에 있던 개념을 Gemini가 **발신자 ID + 반복 납품 시 텍스트 변화**로 구체화. 구현 규칙:

```
발신자 A의 주문 N회 납품 → 텍스트 개인화 단계 상승
미납품/거절 → 서사 체인 중단 or 분기
```

### 1-4. 세금: 계단식 구조
양측 동의. 텍스트 게임 특성상 연속적 누진보다 **등급별 점프**가 판단 포인트로서 명확함.

### 1-5. 전임자 설비 복구 = 초반 마일스톤 체인
양측 동의. 튜토리얼 + 가공 해금 + 로어 도입을 한 흐름에 묶는 구조:

```
공방 탐색 → 고장난 설비 발견 → 재료로 수리 → 설비 해금
                                 └→ 숨겨진 메모 → 파편 로어 + 상위 레시피 힌트
```

단, "메인 퀘스트"가 아니라 **마일스톤 체인**으로 명명 (이 게임에 퀘스트 구조 없음).

### 1-6. 전투: 트레잇 = 행동 가중치 + 조건부 우선순위
Claude의 "행동 선택 가중치" + Gemini의 "IF-THEN 우선순위" 결합:

```
트레잇이 정의하는 것:
  - 행동 선택 확률 (공격적: 공격 80%, 방어 10%, 스킬 10%)
  - 조건부 오버라이드 (심약: HP<30% → 방어/철수 확률 90%로 강제)
  - 패시브 수치 보정 (공격적: 공격력 +15%, 방어력 -10%)
```

### 1-7. 이머전트 이벤트 vs 노드의 등급 구분
양측 동의. 모든 감정적 상황이 노드가 되면 무게감 소실.

```
일상적 감정 사건 → 이머전트 이벤트 (텍스트 연출만)
복합 조건 충족 (예: 호감MAX + 연애루트 + 자금 바닥) → 노드 승격 (비가역적 분기)
```

### 1-8. 육성 이력의 합체 계승
이미 설계에 존재. "업보의 대물림"이라는 프레이밍을 채택하되, 극단적 육성 → 특수 트레잇 획득은 ERA식 조교 시스템이 담당.

## 2. 남은 쟁점 (결정 필요)

### 쟁점 A: 방문자(패러디 캐릭터)의 시스템적 차별화 방식

| | Claude | Gemini |
|---|---|---|
| **입장** | 전용 태그 반대. 트레잇으로 차별화 | 전용 태그 '이계' 부여, 시스템 예외 허용 |
| **근거** | 태그 체계 순도, "패러디 없어도 게임 돌아가야 함" 원칙 | 방문자의 이질감을 시스템으로 체감시켜야 함 |
| **리스크** | 방문자가 너무 평범해질 수 있음 | 예외가 예외를 낳아 규칙 체계 붕괴 |

**내 판단**: Claude 쪽이 설계 원칙과 일관됩니다. 방문자의 차별화는 **트레잇의 독특함**(시간조작, 탄막 등)으로 충분하고, 이 트레잇들이 합체 시 만드는 합성 트레잇이 "이질적 결과물"을 자연스럽게 생성합니다. 태그 체계에 예외를 두는 순간, "이 예외는 어디까지?" 라는 질문이 영원히 따라옵니다.

**단, 절충안 가능**: 방문자 종족 자체가 6종족 중 하나로 합체 테이블에 들어가되, 합체 결과가 **다른 조합보다 분산이 크도록** (즉, 예측 불가능성이 높도록) 확률 테이블을 설계하는 방식. 규칙은 같되 결과의 폭이 다른 것.

### 쟁점 B: 전서(나레이터)의 감정 표현 방식

| | Claude | Gemini |
|---|---|---|
| **입장** | 말줄임표/논평의 점진적 등장 (기존 4단계 타임라인) | 데이터 오염/글리치 연출 |
| **근거** | 판타지 연금술 톤에 맞음 | 메타픽션적 반전 가능성 |

**내 판단**: Claude 쪽이 맞습니다. 이 세계관은 디지털/사이버가 아니라 판타지 연금술입니다. "데이터 손실", "에러 메시지" 같은 표현은 톤 파괴. 전서의 감정은 이미 설계된 **문체의 미세 변화**로 전달하는 것이 이 세계관에서 훨씬 소름끼치는 연출입니다.

다만 Gemini의 "전서를 합체 재료로 쓰려 할 때의 시스템적 저항"이라는 아이디어 자체는 흥미로움. 이것은 글리치가 아니라 **전서가 거부하는 서사적 텍스트**("...이 페이지는 넘어가지 않는다")로 표현하면 톤을 유지하면서 메타적 반전을 넣을 수 있음.

### 쟁점 C: 원소 간 상호작용 규칙의 범위

Gemini가 제안한 원소 간섭(열+위=폭발 증폭, 동+광=분산 부여)은 Claude의 2단계에 들어갈 수 있지만, **규칙의 복잡도**가 문제.

- 원소 5종의 쌍 조합 = 10쌍 (식 제외 4종이면 6쌍)
- 각 쌍에 증폭/상쇄/중립 판정 = 관리 가능한 범위

**내 판단**: 원소 간섭을 넣되, **1단계(기능태그 매트릭스)가 먼저 확정된 후** 2단계로 추가하는 순서를 지켜야 합니다. 동시에 설계하면 복잡도가 곱셈으로 증가.

### 쟁점 D: 세금 노드 분기 (Gemini의 "정치적 노드")

Gemini 제안: 등급 상승 시 "특권 유지 vs 세금 감면" 선택지 → 유닛 영입 제한 or 심층 재료 독점권.

이건 매력적이지만 **NPC 제로 원칙과의 정합성**을 따져야 합니다. "누가" 이 선택지를 주는가? 도시 시스템이 자동으로 제시한다면, 그건 사실상 보이지 않는 NPC(정부)가 있는 것. 이것이 "NPC 제로"의 범위 안인지 밖인지는 **세계관 레벨의 결정**입니다.

**내 판단**: 세금 노드는 넣되, "누군가의 제안"이 아니라 **"규정에 의한 자동 적용"**으로 프레이밍. 즉 "등급 3 도달 시 두 가지 세금 체계 중 선택"이 전서를 통해 규정처럼 전달되는 방식. 이러면 NPC 없이도 작동.

### 쟁점 E: 실행 환경

Claude가 제기했으나 양측 모두 미결정. 이건 브레인스토밍이 아니라 **프로젝트 결정**이 필요한 사항.

| 환경 | 장점 | 단점 |
|------|------|------|
| 터미널 Python | 최속 프로토타입, JSON 자연스러움 | UI 한계, 배포 어려움 |
| 디스코드 봇 | 멀티플레이어, 비동기, 커뮤니티 | DB 필요, API 제약 |
| 웹 | UI 자유도, 접근성 | 프론트 개발 비용 |

## 3. 최종 제안: 다음 단계 우선순위

모든 논의를 종합하면, **다음 작업의 순서**는 이렇게 됩니다:

### Phase 0: 전제 결정 (이것 없이는 아래가 다 공중)
1. **실행 환경 확정** → 세이브 구조, 상태 관리, 이벤트 루프의 물리적 형태가 결정됨

### Phase 1: 조합 시스템 확정 (게임의 심장)
2. **기능태그 10×10 매트릭스 초안** → 어떤 조합이 어떤 카테고리를 만드는지
3. **원소 간섭 규칙 6쌍 정의** (식 제외 4원소 기준) → 2단계 변형자 규칙
4. **형태태그 결합 규칙** → 최종 결과물 형태 결정

### Phase 2: 전투 공식 (조합 결과물이 쓰이는 곳)
5. **턴 순서 / 공격-방어 수식** → 최소 작동하는 전투 루프
6. **트레잇 → 행동 가중치 + 조건부 오버라이드 테이블**

### Phase 3: 경영 곡선 (전투와 조합의 경제적 맥락)
7. **계단식 세율 테이블** (등급별 세율, 유지비 구조)
8. **가격 공식** (아이템 판매가 = 재료 원가 + 가공 보정 + 원소 희귀도)

### Phase 4: 이머전트 이벤트 엔진
9. **이벤트 조건식 포맷 정의** (풀 스캔 방식, 매일 체크)
10. **이벤트 간 우선순위/상호배제 규칙**
11. **이머전트 → 노드 승격 조건**
