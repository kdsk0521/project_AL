# Dual Brain 결과

## 질문
지금 조합명 자체를 어떻게 해야할지 모르겠어. 플레이어가 붙이게할지 아니면 그냥 둘지. 아니면 확실한 조합식만 나오게 여럿만들던가.

---

## Round 1 — 각자 의견

### Claude
이 고민의 핵심은 **자유 조합의 결과물이 너무 많아서 전부 이름을 미리 붙일 수 없다**는 점이죠.

현재 설계에서 선택지를 정리하면:

## 1. 플레이어 네이밍 (현재 설계)

**문제**: 매번 이름 붙이는 게 귀찮아짐. 100번째 조합에서도 이름을 지어야 한다면 작업이 됨. 텍스트 게임에서 UX 부담이 큼.

## 2. 자동 생성 이름 (추천)

태그 조합에서 이름을 **기계적으로 조립**하는 방식.

```
[원소] + [기능] + [형태 기반 카테고리명]

식 + 회복 + 액체  → "식의 회복약"
동 + 공격 + 광물  → "동의 전투검"
위 + 수호 + 유기  → "위의 방호대"
열+동 + 독성 + 분말 → "열동의 독분"
```

- 태그 조합 → 이름 자동 생성 규칙만 만들면 됨
- 플레이어가 원하면 **리네임** 가능 (선택적)
- 같은 태그 조합이면 같은 이름 → "이거 전에도 만들었네" 인지 가능
- tier가 다르면 접두어로 구분: "상급 식의 회복약"

## 3. 확정 레시피만 (축소 방향)

자유 조합을 줄이고, **의미 있는 조합만** 이름+효과가 있는 결과물로.

**문제**: "모든 것은 조합이다"는 코어 문법과 충돌. 레시피 암기 게임이 될 위험.

## 제 의견

**2번(자동 생성) + 특수 레시피는 고정 이름**이 가장 현실적입니다.

이유:
- 자유 조합의 "뭐든 넣으면 뭔가 나온다"를 유지하면서
- 이름 짓는 부담이 없고
- 특수 레시피("기초 회복 물약", "심연의 물약")는 고유 이름으로 발견의 쾌감을 줌
- 자동 이름이 다소 무미건조한 건 오히려 **특수 레시피의 가치를 높이는 효과**

자동 이름 생성 규칙은 태그 체계가 이미 잘 잡혀있으니 거기서 바로 파생 가능합니다. 형태 태그 → 카테고리명 매핑 테이블 하나만 만들면 됩니다:

```
액체 → ~약/~수   분말 → ~산/~분   광물 → ~검/~갑/~석
유기 → ~체/~핵   기체 → ~연/~기   결정 → ~정/~주
식물 → ~환/~초
```

이 방향으로 갈지, 아니면 다른 쪽을 더 파고 싶은지요?

### 1. [절충안] 시스템 제안 + 플레이어 라벨링 (추천)
가장 권장하는 방식입니다. 시스템이 태그를 조합해 **'기술적 명칭'**을 먼저 만들고, 플레이어가 원할 때만 **'고유 이름'**을 붙이게 하는 것입니다.

*   **시스템 작동**: 조합 결과가 나오면 태그를 조합해 이름을 자동 생성합니다. (예: `회복` + `열` + `액체` = **"따뜻한 회복 액체"**)
*   **플레이어 개입**: 플레이어가 "이건 내 시그니처 아이템이야!"라고 느끼면 이름을 바꿀 수 있습니다. (예: **"전임자의 해장술"**)
*   **시스템적 이득**: 플레이어가 이름을 붙이는 순간, 그 **태그 조합은 플레이어의 '개인 레시피 북'에 등록**됩니다. 다음번에 똑같은 태그 조합으로 만들면 자동으로 그 이름이 붙습니다.
*   **서사적 연결**: 가게에 진열할 때 플레이어가 직접 붙인 이름이 노출되면 "자신만의 공방"이라는 느낌이 강해집니다.

### 2. [논리 중시] 서술적 명칭 자동 생성 (MUD 스타일)
플레이어의 입력을 최소화하고, 조합의 결과물을 '관찰'하게 만드는 방식입니다.

*   **방식**: `[상태/원소] + [효과] + [형태]` 순서로 이름을 자동 조립합니다.
*   **예시**: 
    *   `위` + `공격` + `분말` = **"압축된 파괴 가루"**
    *   `동` + `수호` + `결정` = **"진동하는 방어 결정"**
*   **장점**: 직관적입니다. 이름을 보면 이 아이템이 무슨 성질인지 바로 알 수 있습니다. "레시피를 외우는 게임"이 아니라 "성질을 읽는 게임"이 됩니다.
*   **단점**: 모든 아이템 이름이 비슷해 보일 수 있어 수집욕이 저하될 수 있습니다.

### 3. [수집 중시] 티어별 고정 명칭 + 변종 태그
일반적인 RPG 방식이지만, 태그를 '접두사'로 활용합니다.

*   **방식**: 기능 태그 조합으로 **카테고리(기본 이름)**가 정해지고, 원소 태그가 **접두사**가 됩니다.
*   **예시**:
    *   `회복` + `액체` 조합은 무조건 **"회복약"**
    *   여기에 `열`이 섞이면 **"불타는 회복약"**, `식`이 섞이면 **"기이한 회복약"**
*   **장점**: 깔끔하고 관리가 쉽습니다.
*   **단점**: 현재 우리가 지향하는 "모든 것이 조합이고 창발적이다"라는 철학에 비해 조금 딱딱하게 느껴질 수 있습니다.

2.  **도감의 완성**: 플레이어가 이름을 붙이면 그 아이템은 `도감`에 그 이름으로 기록됩니다.

4.  **전서의 논평**: 전서(나레이터)가 플레이어의 네이밍 센스를 보고 한마디 하게 합니다. 
    *   (멋진 이름을 붙였을 때) : *"...격식 있는 이름이다. 기록한다."*
    *   (이상한 이름을 붙였을 때) : *"...정말 이 이름으로 할 건가? 알겠다. 기록한다."*

**결론적으로:** 
"완전 자유 네이밍"은 피로할 수 있으니, **[시스템이 태그 기반으로 '임시 이름'을 제안]** -> **[플레이어가 '확정'하거나 '수정'함]** -> **[확정된 이름은 해당 레시피의 고유명이 됨]** 구조가 가장 시너지(경험의 무한함 + 시스템의 논리)가 좋을 것 같습니다.

어떤 방향이 공방 주인의 감각에 더 가까우신가요? 이 결정에 따라 `system_crafting.md`의 데이터 구조(네이밍 테이블 등)를 손봐야 할 것 같습니다.

## 3가지 네이밍 전략에 대해

**1번(시스템 제안 + 플레이어 라벨링)과 2번(서술적 명칭 자동 생성)은 사실상 같은 것입니다.** 둘 다 "시스템이 태그 기반으로 이름을 자동 생성하고, 플레이어가 원하면 바꾼다"인데 Gemini가 별도 선택지처럼 나열한 것일 뿐입니다. 차이는 자동 생성 이름의 **톤**(기술적 vs 서술적)뿐이고, 이건 네이밍 테이블의 단어 선택 문제지 구조적 차이가 아닙니다.

**3번(티어별 고정 명칭 + 접두사)**도 결국 자동 생성의 변형입니다. "회복약"이라는 카테고리명에 원소 접두사를 붙이는 건 자동 생성 규칙의 한 가지 포맷일 뿐.

즉, Gemini가 3가지 전략이라고 제시한 것은 **실질적으로 1가지 전략의 3가지 포맷**입니다. 구조적 선택지가 아니라 표현 방식의 차이.

## "연금술사의 명명권" 제안에 대해

### 최초 발견자 특권 — 부분 동의

"처음 만들었을 때만 이름을 붙인다"는 **UX 부담을 줄이는 좋은 제한**입니다. 매번 이름을 묻지 않으니까. 다만:

- "최초 발견"의 단위가 **태그 조합 세트**인지, **구체 재료 조합**인지를 정해야 합니다. 약초+물이나 약초+다른액체나 태그 합산이 같으면 같은 "레시피"인가? 현재 설계상 태그 합산 결과가 같으면 같은 결과물이니, **태그 세트 단위**가 맞습니다.
- 그런데 태그 조합 공간이 넓으면 "최초 발견" 자체가 빈번해서 결국 매번 이름을 물어보는 것과 다를 바 없어집니다. 이건 실제 조합 공간의 크기에 따라 달라지는 문제.

### 도감 등록 — 동의, 이미 있는 설계

아이템 도감이 이미 수집 요소로 확정되어 있습니다. 플레이어가 붙인 이름이 도감에 기록되는 건 자연스러운 확장.

### 손님 반응 ("바보 멍청이 약") — **반대**

이건 매력적으로 들리지만 **시스템 비용이 높습니다.**

- 주문서 텍스트에 플레이어 입력 문자열을 삽입하려면, 주문서 생성 로직이 기존 판매 이력의 아이템명을 참조해야 합니다. 단순 플레이버가 아니라 **동적 텍스트 생성**이 필요.
- 플레이어가 부적절한 이름을 붙였을 때 그게 게임 텍스트에 그대로 반영되면 톤이 깨집니다.
- 무엇보다, 이건 **"있으면 좋지만 없어도 게임이 돌아가는" 장식**입니다. 핵심 시스템을 확정하기 전에 논의할 단계가 아닙니다.

### 전서의 논평 — **조건부 동의**

전서가 네이밍에 반응하는 건 전서 자아 복선 연출로 쓸 수 있습니다. 다만:

- 이건 전서 톤 변화 타임라인의 **중반 이후**(레이어 3 논평 단계)에서나 가능합니다. 초반에 전서가 이름에 반응하면 "이게 살아있다"가 너무 빨리 드러납니다.
- "멋진/이상한"을 시스템이 판정하려면 네이밍 품질 판정 로직이 필요한데, 이건 불필요한 복잡도입니다. 하려면 **이름 길이나 특수문자 여부** 같은 단순 규칙으로 한정해야 합니다.
- 역시 **나중에 추가할 수 있는 장식**이지, 지금 확정할 사항이 아닙니다.

### 1. [보완] 자동 생성 로직의 고도화: "서술적 정의"

Claude가 제안한 `[원소] + [기능] + [형태]` 조합은 매우 효율적입니다. 여기에 **'모순 쌍(Contradiction pairs)'** 시스템을 결합하여, 해석 불가능한 결과물에 대한 네이밍 규칙을 추가해야 합니다.

*   **해석 가능 (모순 0~4쌍)**: Claude의 규칙 적용.
    *   예: `식` + `회복` + `액체` → **"식의 회복수"**
*   **해석 불가 (모순 5쌍 이상)**: `system_crafting.md`의 '미지의 결과물' 규칙에 따라 기괴한 접두사 부여.
    *   예: **"일그러진 무언가"**, **"혼돈의 잔해"**, **"이름 붙일 수 없는 액체"**
*   **효과**: 플레이어는 이름을 보고 "이 조합은 연금술적으로 안정적이다/불안정하다"를 즉각 직관할 수 있습니다.

### 2. [추가] "발견자의 명명권": 자동 생성의 건조함 해결

모든 아이템이 "식의 회복수"라면 수집욕이 떨어집니다. 이를 **'최초 발견'** 시스템으로 해결할 것을 제안합니다.

*   **시스템 제안 후 확정**: 처음 보는 태그 조합을 성공했을 때, 시스템은 자동 생성된 이름을 보여주며 묻습니다. 
    *   *"[식의 회복수]가 연성되었습니다. 이 레시피에 고유한 이름을 붙이겠습니까?"*
*   **레시피 북 등록**: 플레이어가 이름을 입력(예: **"월광의 눈물"**)하는 순간, 해당 태그 조합은 도감에 그 이름으로 **영구 등록**됩니다. 이후 동일 조합 연성 시 "월광의 눈물"로 출력됩니다.
*   **서사적 가치**: 이는 플레이어가 시스템(태그)을 정복하여 자신만의 '언어(이름)'로 치환해 나가는 **성취감**을 줍니다.

### 3. [연결] NPC 제로 환경과의 시너지

이름을 플레이어가 붙이게 되면, `system_npc_relationship.md`에서 정의한 **'게시판'** 채널과 강력하게 맞물립니다.

*   **피드백 루프**: 플레이어가 이름을 붙인 아이템이 일정 수 이상 팔리면, 의뢰 게시판의 주문서에 해당 이름이 등장합니다.
    *   *의뢰: "최근 소문난 **'월광의 눈물'** 3병을 구합니다."*
*   **전서의 논평**: Claude가 제안한 전서(나레이터)의 톤 변화와 연결합니다. 
    *   멋진 이름에는 감탄을, 기괴한 이름(예: "빨간물")에는 *"...직관적인 이름이다. 기록한다."* 같은 냉소적인 반응을 출력하여 '지켜보고 있다'는 감각을 줍니다.

## Round 3 — 최종 정리 (Claude)
# 네이밍 시스템 최종 정리

## 1. 합의점

양쪽 모두 동의하는 확정 사항:

- **자동 생성이 기본이다.** 태그 조합에서 이름을 기계적으로 조립. 플레이어에게 매번 이름을 묻지 않음.
- **포맷: `[원소] + [기능] + [형태 카테고리명]`** — 구체적인 단어 선택은 매핑 테이블로 해결.
- **특수 레시피는 고정 이름.** 자동 생성명과 별도로 고유 이름이 있는 "발견형" 레시피 존재.
- **리네임은 선택적으로 허용.** 강제하지 않음.
- **도감 등록.** 플레이어가 만든 조합(+이름)은 도감에 기록됨.
- **동일 태그 세트 = 동일 아이템.** 재료가 달라도 태그 합산 결과가 같으면 같은 결과물.

## 2. 남은 쟁점

| 쟁점 | Claude 입장 | Gemini 입장 | 판정 |
|------|------------|------------|------|
| **리네임 타이밍** | 언제든 가능 (단순) | 최초 발견 시에만 묻기 | **Gemini 쪽이 나음.** 매번 리네임 옵션 노출은 노이즈. 최초 발견 시 한 번만 묻고, 이후엔 도감에서 수정 가능하게 하면 UX 부담 최소화 + 발견 쾌감 확보. 단, 조합 공간이 넓어서 "최초 발견"이 너무 빈번하면 역효과 — 이건 실제 태그 조합 수를 세봐야 알 수 있음. |
| **주문서에 플레이어 이름 반영** | 반대 (시스템 비용, 톤 파괴) | 찬성 (피드백 루프) | **Claude가 맞음.** 동적 텍스트 삽입은 비용 대비 효과 낮고, 부적절한 이름이 게임 텍스트를 오염시킬 위험. 주문서는 시스템 생성명이나 고정 레시피명으로 출력하는 게 안전. |
| **전서 네이밍 논평** | 나중에 (전서 타임라인 중반 이후) | 초반부터 가능 | **Claude가 맞음.** 전서의 자아 드러남은 단계적으로 진행되어야 서사적 임팩트가 있음. 초반부터 이름에 반응하면 복선이 너무 일찍 노출됨. 레이어 3(논평 단계) 이후 장식으로 추가하는 게 적절. |
| **모순 조합의 이름** | 별도 언급 없음 | "일그러진 무언가" 등 기괴한 접두사 | **Gemini 아이디어 채택 가치 있음.** 다만 "모순 5쌍 이상"같은 수치 기준은 실제 모순 쌍 시스템이 확정된 후에 정해야 함. 원칙만 확정: 해석 불가 결과물은 자동 생성명 대신 별도 불안정 네이밍 풀에서 뽑는다. |

## 3. 최종 제안

### 확정할 구조

```
[정상 조합]
  자동 생성명 = 원소 키워드 + 기능 키워드 + 형태 접미사
  예: "식의 회복수", "동의 전투검", "열동의 독분"
  
  티어 구분: 접두어 ("상급 ~", "정제된 ~")

[특수 레시피]  
  고정 고유명 ("심연의 물약", "기초 회복 물약")
  자동 생성명 대신 고유명 출력

[불안정 조합 (모순 과다)]
  별도 네이밍 풀: "일그러진 무언가", "이름 없는 잔해" 등
  
[리네임]
  최초 발견 시 1회 제안 → 수락/거절
  거절 시 자동 생성명 유지
  이후 도감에서 언제든 수정 가능
```

### 즉시 필요한 작업

1. **형태 태그 → 접미사 매핑 테이블 확정** (액체→수/약, 분말→산/분, 광물→검/갑/석 등)
2. **원소 태그 → 접두 키워드 매핑 테이블 확정** (식→식의, 열+동→열동의 등)
3. **기능 태그 → 중간 키워드 매핑 테이블 확정** (회복, 공격→전투, 수호→방호 등)
4. **불안정 네이밍 풀** (10~15개 정도의 기괴한 이름 목록)

이 4개 테이블이 확정되면 네이밍 시스템은 완성입니다. 나머지(전서 논평, 주문서 반영 등)는 전부 이 위에 나중에 얹는 장식 레이어입니다.
