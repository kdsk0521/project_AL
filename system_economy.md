## 개요

경영 시스템은 조합/제작 시스템 위에 얹히는 **수익화 레이어**.
만든 것을 팔아서 돈을 벌고, 번 돈으로 설비를 늘리고, 설비가 제작 범위를 확장하는 순환.

```
[조합/제작] → 아이템/유닛 생산
       ↓
[판매] → 자금 획득 (태그/능력치가 가격을 결정)
       ↓
[투자] → 설비 업그레이드, 도구 제작, 탐사 비용
       ↓
[확장] → 더 깊은 던전, 더 넓은 제작 범위
       ↓
[조합/제작] → 더 좋은 아이템/유닛 생산
       ...순환
```

---

## 설계 원칙

> **"경영의 수익은 시세가 아니라 실력이 결정한다."**

- 수요/시세 변동 시스템 **없음**. 복잡한 경제 시뮬레이션은 지양
- 아이템의 가격은 **태그, 등급(tier), 효과 수치**에 의해 결정
- 유닛의 가격은 **스탯, 특성, 육성 이력**에 의해 결정
- 경영의 재미 = "이 손님이 원하는 걸 얼마나 잘 맞출 수 있는가"
- 조합/육성을 잘할수록 수익이 올라가는 구조 → 시스템 간 자연스러운 연결

## 판매 시스템

### 2채널 판매 구조

#### 1. 가게 진열 (수동 판매)

제작한 아이템을 가게에 진열해두면 자동으로 팔린다.

- **기본 가격**으로 판매 (태그, tier 기반 자동 산정)
- 진열 슬롯 제한 (가게 규모에 따라 확장)
- 특별한 보너스 없이 안정적 수입
- "만들어두고 진열" → 기본 자금 흐름의 안전망

```json
{
  "item_id": "ITEM_MANA_FOOD_T2",
  "name": "2등급 마나회복 음식",
  "tags": ["마나회복", "영양"],
  "tier": 2,
  "base_price": 150,
  "sale_type": "display"
}
```

#### 2. 손님 주문 (능동 판매)

손님이 **특정 조건의 아이템/유닛**을 요구하며 찾아온다.
조건을 맞출수록 보너스가 붙는다.

```json
{
  "order_id": "ORD_001",
  "customer": "여행자",
  "request": {
    "category": "consumable_potion",
    "required_tags": ["회복"],
    "preferred_tags": ["강화회복", "보존"],
    "min_tier": 2
  },
  "base_reward": 200,
  "tag_bonus": {
    "per_preferred_tag": 50,
    "all_preferred_bonus": 150
  },
  "tier_bonus": {
    "per_tier_above_min": 30
  },
  "deadline": 5,
  "lore": "먼 길을 떠나는 여행자가 든든한 회복약을 찾고 있다."
}
```

#### 주문 보너스 계산 예시

```
주문: 회복 물약, 최소 tier 2, 선호 태그 [강화회복, 보존]

납품: 3등급 강화회복 보존 물약 (tier 3, 태그: 강화회복 + 보존 + 회복)

보상 계산:
  기본 보상            200
  + 선호 태그 2개 매칭  100 (50 × 2)
  + 전체 선호 달성 보너스 150
  + tier 초과 1단계     30
  = 총 480
```

- 필수 태그를 못 맞추면 납품 불가
- 선호 태그는 맞출수록 보너스
- tier가 높을수록 추가 보상
- **손님이 원하는 걸 정확히 읽고, 조합으로 맞추는 것이 경영의 핵심 판단**

### 유닛 판매

유닛도 판매 대상이 된다. 가격은 스탯, 특성, 육성 이력에 의해 결정.

```
유닛 판매 가격 = 기본 종족가 + 레벨 보정 + 스탯 보정 + 특성 보너스 + 육성 이력 보너스
```

- 잘 키운 유닛일수록 비싸게 팔림
- 특정 특성을 요구하는 손님 주문도 발생
  - "공격형 유닛 원함, 선호: 화속성, 연속공격 특성"
- 호감도가 높은 유닛을 팔면 → 관계 시스템에 영향 (이머전트 이벤트 트리거)
- **"이 유닛을 팔 것인가 남길 것인가"가 감정적 판단이 되는 구조** (ERA 참고)

## 가격 결정 구조

시세 변동이 없으므로, 가격은 **아이템/유닛 자체의 속성**이 결정한다.

### 아이템 가격 공식 (초안)

```
기본 가격 = 카테고리 기본값 × tier 배수
태그 가산 = 효과 태그 수 × 태그당 가산값
최종 가격 = 기본 가격 + 태그 가산
```

- 카테고리별 기본값: 소비재 < 도구 < 장비 < 상위 소재
- tier가 높을수록 배수가 올라감
- 태그가 많이 실린 아이템일수록 비쌈 → 조합 숙련의 보상
- 구체적 수치는 밸런싱 단계에서 확정

### 유닛 가격 공식 (초안)

```
기본 가격 = 종족 기본값 × 레벨 배수
스탯 가산 = 주요 스탯 합계 × 보정 계수
특성 가산 = 보유 특성 수 × 특성 등급별 가산값
육성 이력 가산 = 특수 육성 경로 보너스 (희소한 방식으로 키웠을수록 높음)
최종 가격 = 기본 가격 + 스탯 가산 + 특성 가산 + 육성 이력 가산
```

## 설비/시설 시스템

경영의 성장감은 **거점(공방/가게)의 확장**에서 온다.

### 설비 카테고리

| 설비 유형 | 역할 | 예시 |
|----------|------|------|
| 가공 설비 | 가공 방식 해금 / 가공 범위 확장 | 건조대, 가마, 분쇄기, 발효통, 정제기 |
| 제작 설비 | 조합 가능 카테고리/등급 확장 | 기초 작업대 → 고급 작업대 → 마법 작업대 |
| 가게 설비 | 진열 슬롯 수, 손님 주문 빈도 | 가판대 → 소규모 가게 → 대형 가게 |
| 저장 설비 | 인벤토리/창고 용량 | 상자 → 창고 → 대형 창고 |
| 특수 설비 | 고유 기능 해금 | 매크로 작업대, 연구실 등 (향후 확장) |

### 설비 업그레이드 구조

- 설비는 **조합으로 제작** (제작 도구 카테고리)
- 설비 등급이 올라가면:
  - 새로운 가공 경로 개방
  - 상위 tier 아이템 제작 가능
  - 진열 슬롯 증가, 손님 주문 등급 상승
  - 태그 슬롯 상한 확장 (자유도 단계적 개방)
- 설비 업그레이드 자체가 **마일스톤 트리거**가 될 수 있음

### 설비와 다른 시스템의 연결

```
[탐사] → 설비 제작 재료 획득
[조합] → 설비 제작 (도구 카테고리)
[경영] → 설비 설치 → 가공/제작 범위 확장 → 더 좋은 아이템 → 더 높은 수익
[관계] → 특정 유닛이 설비 효율 보너스를 줄 수 있음 (미결정)
```

## 자금 순환

### 수입원

- **가게 진열 판매**: 안정적, 저수익
- **손님 주문 납품**: 변동적, 고수익 (조건 매칭 정도에 따라)
- **유닛 판매**: 고수익, 비정기 (감정적 비용 수반)
- **특수 거래**: 마일스톤/노드 분기에서 발생하는 특별 거래 기회

### 지출처

#### 고정 지출 (가만히 있어도 나가는 돈 — 자금 압박의 핵심)

- **세금**: 공방/가게 규모에 비례. 설비를 늘리고 가게를 키울수록 세금이 올라감
  - 성장에 비용이 따르는 구조 → "지금 확장할 여유가 있나?" 판단
  - 일정 주기(턴/일)마다 자동 차감
- **유닛 유지비**: 보유 유닛 수와 등급에 비례. 많이 데리고 있을수록 비용 증가
  - "이 유닛을 계속 먹여살릴 것인가" 판단 → 유닛 판매 압박과 연결
  - 유닛이 많을수록 탐사 편성 선택지는 넓지만 유지비가 경영을 압박

#### 선택 지출 (플레이어 판단에 의한 투자)

- **탐사 비용**: 던전 진입 시 소비 아이템, 장비 소모
- **설비 제작/업그레이드**: 공방 확장의 핵심 투자
- **도구 제작**: 채집/육성/제작 도구 업그레이드
- **유닛 획득**: 새 유닛 영입 비용
- **기타**: 재료 구입 (시스템 상점에서 기초 재료)

### 자금 흐름의 긴장감

- **고정 지출이 존재하므로 가만히 있으면 돈이 줄어든다** → 계속 팔아야 함
- 확장하면 수입은 올라가지만 세금도 올라감 → "더 많이 벌어야 유지된다"
- 유닛을 많이 데리면 탐사는 유리하지만 유지비가 경영을 압박 → 적정 규모 판단
- 수입과 지출의 균형이 항상 팽팽해야 "뭘 먼저 투자할까"의 판단이 생김
- 자금 부족 → 이머전트 이벤트: "유닛을 팔아야 하나?", "긴급 탐사 가야 하나?"
- 이 긴장감이 경영 시스템의 재미이자 다른 시스템과의 간섭 지점

## 시간 구조

### 기본 단위: 월 (프린세스 메이커 참고)

게임의 시간은 **월 단위**로 흐른다.
한 달 안에서 플레이어는 스태미나 총량 내에서 자유롭게 행동한다.

```
[월 시작]
  - 이번 달 손님 주문 갱신
  - 달력 이벤트 확인 (고정 이벤트, 시즌 이벤트)
  - 자금/재고 상태 확인
       ↓
[매일 — 하루 단위 행동]
  - 하루 스태미나 내에서 행동 선택
  - 탐사, 가공, 조합, 판매, 육성, 가게 관리 등
  - 행동별 스태미나 소모량이 다름
  - 스태미나 소진 or 자발적 종료 → 하루 종료
  - 다음 날 스태미나 리셋
       ↓
[월 말 (한 달치 일수 경과)]
  - 고정 지출 차감 (세금 + 유닛 유지비)
  - 손님 주문 기한 체크 (미납품 시 패널티/기회 소실)
  - 이벤트 판정 (이머전트 이벤트, 마일스톤 체크)
       ↓
[다음 월 시작] → 반복
```

### 스태미나

- 플레이어의 **1일 행동량**을 제한하는 자원
- 매일 리셋 (하루가 끝나면 다음 날 전량 회복)
- 행동별 소모량 차등:
  - 탐사: 높음 (던전 깊이에 따라 추가 소모)
  - 조합/가공: 중간
  - 판매/진열/관리: 낮음
  - 육성: 중간~높음
- 하루치 스태미나로 한두 가지 큰 행동 or 여러 가지 작은 행동을 배분
- **매일의 선택이 곧 한 달의 결과**: "오늘은 탐사" "오늘은 제작" "오늘은 육성"
- 스태미나 총량 확장 방식 (미결정: 설비? 아이템? 레벨?)

### 달력 시스템

월 단위 시간 구조 위에 **달력**이 존재한다.
1년 = 12개월

#### 고정 이벤트 (시즌 이벤트)

달력의 특정 월에 매년 반복되는 이벤트가 배치된다.

- **시즌 수요 변동**: 특정 시기에 특정 카테고리 아이템 수요 상승
  - 예: 겨울 → 보온/회복 계열 수요 증가, 축제 시즌 → 고급 아이템 수요 증가
  - 시세 변동이 아닌 **손님 주문 내용이 시즌에 따라 달라지는** 구조
- **축제/행사**: 특수 손님 출현, 한정 거래 기회, 보너스 보상
- **계절 던전**: 특정 시기에만 열리는 던전 구역이나 채집 가능 재료
- **연중 이벤트**: 기념일, 유닛 관련 이벤트 (생일 등 — 관계 시스템 연결)

#### 달력과 다른 시스템의 연결

```
[달력 → 경영] 시즌별 손님 주문 변화 → "이번 달엔 뭘 만들어야 잘 팔릴까" 판단
[달력 → 탐사] 계절 한정 재료/던전 → "이 시기에만 갈 수 있는 곳" 탐사 동기
[달력 → 관계] 유닛 이벤트 → 호감도 상승 기회
[달력 → 이머전트] 시즌 조건이 이벤트 트리거에 추가됨
```

- 달력이 있으면 시세 변동 없이도 **시기에 따라 최적 판단이 달라짐** → 설계 철학 부합
- 고정 이벤트는 예측 가능하므로 "다음 달 축제를 위해 미리 재고 쌓아두자" 같은 선행 계획이 가능
- 선행 계획 + 이머전트 이벤트의 방해 = 매번 다른 경험

## 손님 시스템

### 손님 유형 (초안)

- **일반 손님**: 기초적인 주문. 필수 태그만 있으면 납품 가능
- **까다로운 손님**: 선호 태그가 많고 tier 요구가 높음. 보상도 높음
- **긴급 주문**: 짧은 기한, 높은 보상. 재고가 있으면 즉시 대응 가능
- **특수 손님**: 마일스톤/노드 분기에 의해 출현. 고유 보상이나 이벤트 체인 시작
- **유닛 구매자**: 특정 조건의 유닛을 요구

### 손님 출현 규칙

- 가게 등급에 따라 손님 풀이 달라짐 (단계적 개방)
- 초기: 일반 손님 위주
- 가게 확장 후: 까다로운 손님, 긴급 주문 추가
- 마일스톤 달성 후: 특수 손님 출현
- **주문 내용에 로어 텍스트 부착** → 파편 로어와 연결 가능

## 경영과 다른 시스템의 간섭

```
[경영 → 탐사]
  - 자금으로 탐사 장비/소비재 준비
  - 가게에서 필요한 아이템이 탐사 목표를 결정 ("이 주문 맞추려면 저 재료가 필요해")

[경영 → 조합]
  - 설비 등급이 제작 가능 범위를 결정
  - 손님 주문이 "뭘 만들까"의 방향을 제시

[경영 → 관계]
  - 유닛 판매 = 관계 변동
  - 자금 부족 → 유닛 해고/판매 압박 → 이머전트 이벤트
  - 특정 유닛이 가게 운영에 보너스를 줄 수 있음 (미결정)

[경영 ← 탐사]
  - 던전에서 얻은 재료가 제작/판매의 원료
  - 탐사 실패(부상/소실)가 경영에 타격

[경영 ← 관계]
  - 호감도 높은 유닛이 경영 보너스 (미결정)
  - 유닛 이탈 시 전력/생산력 감소
```
